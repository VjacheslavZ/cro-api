Run a Prisma schema migration safely — without dropping tables or losing existing data.

Migration name argument: $ARGUMENTS (e.g. "add_matching_item"). If not provided, derive a name from the schema diff.

---

## HARD RULES — Never violate these

- **NEVER run `prisma migrate reset`** — this drops and recreates the entire database, deleting all data
- **NEVER run `prisma db push --force-reset`** — same effect
- **NEVER run `prisma db push`** at all — it bypasses the migration history and can cause schema drift
- **NEVER run raw `DROP TABLE` or `TRUNCATE`** SQL directly
- **NEVER add a `NOT NULL` column without a `@default(...)` value** — this will fail on tables with existing rows

If any of these are needed, STOP and ask the user explicitly before proceeding.

---

## Phase 1 — Analyze (always run this first)

### 1a. Show the schema diff

```bash
git diff -- 'apps/api/src/prisma/schema.prisma'
```

### 1b. Generate the SQL without applying it

```bash
npx prisma migrate dev \
  --schema=apps/api/src/prisma/schema.prisma \
  --create-only \
  --name $ARGUMENTS
```

This creates the migration SQL file in `apps/api/src/prisma/migrations/` but does **not** apply it to the database.

### 1c. Read and analyze the generated SQL

Read the newly created migration file (`migrations/<timestamp>_$ARGUMENTS/migration.sql`) and check for any of these destructive patterns:

| Pattern in SQL | Risk | Action |
|---|---|---|
| `DROP TABLE` | All rows in that table are deleted | **STOP** — confirm with user, suggest soft-delete or rename instead |
| `DROP COLUMN` | All data in that column is deleted | **STOP** — confirm with user; if renaming, write a safe rename migration manually |
| `ALTER COLUMN ... SET NOT NULL` (no default) | Fails if existing rows have NULL | **STOP** — add `@default(...)` in schema or backfill existing rows first |
| `ALTER COLUMN` type change | May truncate values | **STOP** — confirm with user |
| `TRUNCATE` / `DELETE FROM` | Row data lost | **STOP** — never auto-generated by Prisma, but flag if present |

**Column rename detection**: Prisma has no native rename concept — it generates `DROP COLUMN` + `ADD COLUMN`. If the schema diff shows a field renamed (old field removed, new field added with same type), STOP and inform the user. The correct approach is a manual migration:
```sql
-- safe rename: no data loss
ALTER TABLE "TableName" RENAME COLUMN "old_name" TO "new_name";
```
Replace the auto-generated SQL with this before proceeding to Phase 2.

## Phase 2 — Apply (only after Phase 1 confirms no data loss)

### 2a. Apply the migration

```bash
npx prisma migrate dev \
  --schema=apps/api/src/prisma/schema.prisma
```

This applies the already-created migration file from Phase 1 (Prisma detects it's already been generated and applies without re-creating).

### 2b. Regenerate the Prisma client

```bash
npm run -w cro-api prisma:generate
```

### 2c. Seed check

The seed script (`apps/api/src/prisma/seed.ts`) is safe — all inserts use `findFirst`/`findUnique` + skip-if-exists. Run it only if:
- A new enum value was added that seed data depends on, OR
- New required lookup data was added (e.g. a new default topic)

```bash
# Only if needed:
npm run -w cro-api seed
```

### 2d. TypeScript check

```bash
npm run -w cro-api typecheck
```

- Whether seed was re-run and why (or why not)
